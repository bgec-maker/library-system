## 0. 5,000권이 바꾸는 것

권수가 수백일 때 통하던 방식(시트 전체 읽고 JS로 필터)이 여기서 무너집니다. 아래 세 가지가 이 설계의 뼈대입니다.

### ① 도서ID = 행 번호 (O(1) 조회)

**도서ID를 1부터 순차 부여하고, 시트 행 번호와 1:1로 묶습니다.**

```
행 번호 = 도서ID + 1   (1행은 헤더)

도서ID 000123  →  Books 시트 124행
```

스캔할 때마다 5,000행을 뒤질 필요가 없습니다. `getRange(124, 1, 1, N)` 한 번이면 끝. 학생도 동일 규칙.

**대가로 지켜야 할 규칙 (절대):**

| 금지 | 이유 |
|---|---|
| 🚫 **시트 정렬(Sort)** | ID↔행 매핑이 통째로 깨짐. **가장 위험한 함정.** |
| 🚫 행 삽입/삭제 | 이후 모든 ID가 밀림 |
| 🚫 ID 재사용 | 폐기해도 번호는 영구 소멸 |

- 폐기·전출은 **삭제가 아니라 `상태` 변경**입니다.
- 사서가 정렬하고 싶어하는 건 당연합니다 → **필터 보기(Filter views)** 를 쓰세요. 원본 순서를 안 건드립니다.
- Books·Students 시트는 **보호(Protected range)** 를 걸고, 정렬 금지를 README 탭 최상단에 크게 적어두세요.

### ② 활성 / 이력 분리

대출 기록을 한 시트에 계속 쌓으면 몇 년 안에 수만 행이 되고, 그때부터 모든 조회가 느려집니다.

```
Loans        = 지금 대출 중인 것만        (수백 행. 반납하면 여기서 빠짐)
History_YYYY = 반납 완료된 기록 (연도별)   (영구 보존, append-only)
```

- **"이 책 대출중?"** → `Books[행].상태` O(1) 조회. 스캔 없음.
- **"이 학생 몇 권?"** → `Students[행].현재대출수` O(1) 조회.
- **연체 목록** → `Loans`만 훑음 (수백 행).
- **통계** → `History_*`를 야간 트리거가 배치 계산 → `Stats` 시트에 저장. 사용자 요청 중에는 절대 History를 읽지 않습니다.

**반납 처리 순서 (순서가 중요):**
1. `History_YYYY`에 append
2. `Loans`에서 해당 행 삭제
3. `Books.상태` → `보관중`, `Students.현재대출수` −1

2·3이 실패해도 1이 남아 있으므로 **재동기화**로 복구됩니다.

### ③ 카탈로그 미러 (검색)

5,000권을 서버에서 검색하면 매번 느리고 할당량을 태웁니다.

- GAS가 전체 도서를 압축 JSON으로 만들어 **Drive 파일**로 저장 (CacheService는 키당 100KB 제한이라 부족합니다).
- `Config.catalogVersion`을 도서 변경 시마다 증가.
- 클라이언트는 버전만 물어보고(초경량), 바뀌었을 때만 통째로 받아 **IndexedDB**에 캐시.
- **검색은 100% 클라이언트에서.** 즉시 응답 + 오프라인 동작 + 서버 할당량 0.
- 5,000권 ≈ 600KB~1MB (gzip 후 ~200KB). 문제없습니다.

### ④ 6분 벽

GAS 실행은 **6분에 강제 종료**됩니다. 다음 작업은 반드시 배치로 쪼개세요.

- 명렬표 일괄 등록, 5,000권 임포트, 진급·졸업 처리, 라벨 PDF 생성, 통계 재계산

**패턴:** 200건 처리 → 커서를 `PropertiesService`에 저장 → `{done:false, cursor}` 반환 → 클라이언트가 재호출. 진행률 표시.

### ⑤ 할당량 — 무료 Gmail로 충분합니다

무료 Gmail 계정의 스크립트 실행시간은 **하루 90분(5,400초)** 입니다. 이건 *사람이 앱을 쓰는 시간*이 아니라 *구글 서버가 코드를 돌리는 시간*이며, **모든 사용자가 하나의 할당량을 공유**합니다 (앱이 개발자 권한으로 실행되므로).

실제 소요를 계산하면:

| | 코드 실행시간 |
|---|---|
| 대출·반납 200~300건 (건당 ~2초) | 7~10분 |
| **검색** | **0초** ← 카탈로그 미러 덕분에 브라우저에서 처리 |
| 로그인·조회 등 | 5~10분 |
| 야간 통계 배치 | 2~3분 |
| **하루 합계** | **20~30분 (90분의 1/3)** |

**5,000권 초기 등록도 문제없습니다.** 권당 코드 실행은 ~3초지만, 사람이 하루에 등록 가능한 건 200~300권이 한계(스캔·확인·라벨·부착·배가)이므로 하루 15분 정도만 씁니다. **사람 손이 서버보다 느립니다.**

> ⚠️ **진짜 위험은 버그 폭주입니다.** 무한 루프나 재시도 폭주는 90분을 몇 분 만에 태우고, **경고 없이 앱 전체가 다음 날까지 멈춥니다.** → 실행 로그를 시트에 남기고, 재시도 횟수에 상한을 두세요.
>
> ⚠️ **계정 소유권 — 더 중요합니다.** 개인 Gmail 하나가 학생 개인정보 전부를 소유하게 됩니다. 담당자가 그만두거나 계정이 잠기면 도서관이 멈춥니다. → **프로젝트 전용 Gmail 계정을 만들고 기관이 소유**하세요 (비밀번호·복구 정보를 기관 책임자가 관리). 개인 계정 사용 금지.
>
> 💡 정식 인가 학교라면 **Workspace for Education Fundamentals**(무료, 6시간 티어), 비영리 법인이라면 **Workspace for Nonprofits**(무료)를 신청할 수 있습니다. 하지만 위 계산대로 **없어도 무방**합니다.

## 3. 데이터 무결성 3원칙

### ① 멱등키 (`opId`) — 오프라인 큐를 만드는 순간 필수

모든 쓰기 요청에 클라이언트가 UUID를 붙입니다. 서버는 이미 처리한 `opId`면 무시하고 이전 결과를 반환.

**없으면 반드시 터집니다:** 모바일 타임아웃 → 사용자 재시도 → **대출 기록 중복 생성**.

### ② `LockService` — 동시 쓰기

모든 쓰기 함수를 `LockService.getScriptLock()` (timeout 10s)으로 감쌉니다. **ID 발번도 락 안에서.**

### ③ 재동기화 (`admin.resync`)

`Loans`(활성) 기준으로 `Books.상태`와 `Students.현재대출수`를 **전부 다시 계산**합니다.

수기로 시트를 손댄 뒤, 또는 처리가 중간에 실패한 뒤 반드시 누르는 버튼. 수백 행만 읽으면 되므로 몇 초면 끝납니다.

## 4. 권한 매트릭스

> **핵심 뒤집기: 셀프 대출은 허용, 셀프 반납은 금지.**
>
> 대출은 책임을 만들고, 반납은 책임을 지웁니다. 지우는 쪽이 위험합니다.
> 셀프 반납을 열면 학생이 집에서 "반납"을 누르고 책을 계속 가질 수 있고, 시스템은 그 책을 서가에 있다고 믿습니다 — **장서점검 때까지 아무도 모릅니다.**
> 반면 셀프 반납의 편의성 이득은 **0**입니다. 책을 돌려주려면 어차피 도서관에 와야 하니까요.

| 기능 | 관리자 | 사서·교사 | 스테이션 | 학생 |
|---|:---:|:---:|:---:|:---:|
| **인증 방식** | Google | Google | **기기 토큰** | 학번+PIN |
| 대출 | ✅ | ✅ | ✅ | ✅ *(셀프)* |
| **반납** | ✅ | ✅ | ✅ | ❌ |
| 연장 | ✅ | ✅ | ✅ | ✅ |
| 도서 검색 | ✅ | ✅ | ✅ | ✅ |
| **학생 목록 검색** | ✅ | ✅ | ❌ | ❌ |
| 방금 스캔한 학생 정보 | ✅ | ✅ | ⚠️ *순간 표시만* | — |
| 내 대출·기록 | — | — | — | ✅ *(본인만)* |
| 도서 등록·수정·폐기 | ✅ | ✅ | ❌ | ❌ |
| 학생 등록·수정 | ✅ | ✅ | ❌ | ❌ |
| 연체 목록 | ✅ | ✅ | ❌ | ❌ |
| 장서점검 | ✅ | ✅ | ✅ | ❌ |
| 한줄평 작성 | — | — | — | ✅ |
| 한줄평 승인·삭제 | ✅ | ✅ | ❌ | ❌ |
| 통계·보고서 | ✅ | ✅ | ❌ | 랭킹만 |
| 설정·계정·기기 발급 | ✅ | ❌ | ❌ | ❌ |
| 일괄 진급·졸업 | ✅ | ❌ | ❌ | ❌ |

**스테이션의 개인정보 노출을 반드시 좁히세요.** 방금 스캔한 학생의 이름·학반·현재 대출 목록만 **순간 표시**하고, 학생 목록 검색은 불가. 도우미가 반 친구 독서기록을 뒤지는 걸 막는 선입니다.

**셀프 대출 차단 조건:** 연체 중 / 한도 초과 / 예약된 책 / 참고도서. 기록에 `처리자 = self`로 남겨 분쟁 시 구분.

**PIN의 진짜 위험은 대출이 아니라 개인정보입니다.** 친구 PIN을 알아내도 최악이 "책 한 권 빌리기"면 괜찮지만 "친구 독서기록 열람"이면 심각합니다. 그래서 **다른 학생을 조회하는 API 자체를 만들지 마세요.** 랭킹·한줄평은 닉네임으로.

⚠️ **학생증 QR로 로그인시키지 마세요.** QR은 신원 *표시*일 뿐 *인증*이 아닙니다 (사진 찍으면 복제됨). 사람이 지켜보는 스테이션에서만 유효합니다.

## 5. GAS API 함수 목록

전부 `POST /exec` 하나로 받고 `{action, payload, opId, token}`으로 분기합니다.
**권한 검증은 반드시 서버 함수 안에서.** 프론트에서 버튼 숨기는 건 UI일 뿐 보안이 아닙니다.

### 인증
```
auth.login(googleIdToken | {studentId, pin} | {deviceCode})  → session
auth.session()                                               → {role, id, name, perms}
auth.logout()
```
- Google ID 토큰은 `tokeninfo`로 검증 → `aud`·`email_verified`·`hd`(학교 도메인) 확인 → `Users` 시트 대조
- 검증 후 **세션 토큰 발급 (CacheService, 6~12h)**. 매 요청 검증하면 UrlFetch 할당량이 터집니다.

### 스캔 · 대출 · 반납
```
scan.resolve(value)                    → {type:'book'|'student', id, ...}
loan.checkout({bookId, studentId, opId})
loan.return({bookId, opId})            → {overdueDays, pointsEarned}
loan.renew({loanId, opId})
loan.activeByStudent(studentId)
loan.activeByBook(bookId)
loan.overdue({grade?, class?})
```
> `scan.resolve`가 QR 값을 받아 책인지 학생인지 판별합니다. **대출/반납을 UI에서 나누지 마세요** — 책을 스캔하면 상태를 보고 자동 판단하면 됩니다 (`대출중`→반납, `보관중`→"누가 빌리나요?"). 애매함이 없어 오작동 위험도 없습니다.

### 카탈로그
```
catalog.version()   → {v}          // 초경량. 클라이언트가 자주 호출
catalog.dump()      → {url}        // Drive JSON. 버전 바뀔 때만
catalog.book(id)
```

### 도서
```
book.lookupISBN(isbn)      → 서지 (외부 API + BookCache)
book.create({...})         → bookId
book.update(bookId, {...})
book.discard(bookId, reason)
book.labelQueue()          → 라벨 미출력 목록
book.labelPrinted(ids)
```

### 학생
```
student.create({...})
student.bulkCreate(rows, cursor)   → {done, cursor, progress}   // 배치
student.update(id, {...})
student.promote(cursor)            // 진급 배치
student.graduate(cursor)           // ⚠️ 미반납 도서 자동 체크
student.resetPin(id)
student.me()                       // 학생 본인용
```

### 독서기록 · 게이미피케이션
```
review.create({bookId, rating, oneLiner, essay})
review.feed({bookId? | grade?})    // 승인된 것만
review.moderate(id, action)
points.my() / points.ranking(scope, period)
bingo.my() / badge.my()
```

### 예약 · 장서점검
```
reserve.create(bookId) / reserve.cancel(id) / reserve.queue(bookId)

inventory.start()                  → sessionId
inventory.scan(sessionId, bookIds[])   // 배치 업로드
inventory.finish(sessionId)        → 누락 목록
```

### 관리
```
admin.config.get() / admin.config.set(k, v)
admin.users.upsert() / admin.users.disable()
admin.device.issue({alias, type, perms, expiry})  → 6자리 코드
admin.device.revoke(id)
admin.resync()                     // ⭐ Books.상태 + Students.현재대출수 재계산
admin.backup()
stats.dashboard() / stats.monthly(ym) / stats.report(period)
```

## 6. 0단계 — 여기가 진짜 프로젝트입니다

맨땅에서 5,000권. 냉정하게 계산해봅시다.

| 항목 | |
|---|---|
| ISBN 있는 책 (자동 조회) | 약 30~45초/권 (스캔·확인·라벨출력·부착·배가) |
| 4,000권 × 40초 | **≈ 45시간** |
| ISBN 없는 책 (구간·기증본·오래된 책, 보통 10~20%) | 2~3분/권 |
| 1,000권 × 2.5분 | **≈ 42시간** |
| **합계** | **≈ 85시간** |

혼자 하면 두 달, 다섯 명이 붙어도 각 17시간입니다. **"다 정리하고 오픈한다"는 계획은 실패합니다.**

### 유일하게 현실적인 전략: 점진적 등록

> **"대출할 때 등록한다."**

- 앱을 **미등록 상태로 오픈**합니다.
- 학생이 등록 안 된 책을 가져오면 → 그 자리에서 30초 등록 → 바로 대출.
- **인기 있는 책부터 자연스럽게 등록됩니다.** 아무도 안 빌리는 책은 나중에 천천히.
- 병행: 서가 한 구역씩 정복 (완료 구역 표시), 도우미 학생 등록 스테이션 운영.

### ISBN 자동입력이 승부처

수기 입력하면 죽습니다. **폰 카메라로 책 뒤 ISBN 바코드 → API 조회 → 서지 자동완성 → 발번 → 라벨 큐.**

- 국립중앙도서관 서지 API (무료)
- **알라딘 API — 표지 + `페이지수`** ← 게이미피케이션 가중치에 필수
- 네이버 책 API (보조)
- 조회 결과는 `BookCache` 시트에 저장 (복본 재조회 무료)

**⭐ ISBN 조회는 GAS가 아니라 브라우저에서 하세요.**

```
[브라우저] ISBN 스캔 → 도서 API 직접 호출 → 서지 확보 → 사서가 확인·보정
        → GAS에는 "완성된 데이터"만 100건씩 묶어서 전송 (쓰기만)
```

GAS 실행시간이 1/5로 줄고, UrlFetch 할당량 소모가 **0**이 됩니다.
API가 CORS를 안 열어주면 **Cloudflare Worker를 프록시로** (무료, 10만 req/일, ~20줄). 덤으로 API 키가 프론트에 노출되지 않습니다.

### 라벨

- **QR + 등록번호 + 청구기호를 한 라벨에.** QR만 있으면 시스템이 죽었을 때 아무것도 못 합니다.
- QR 최소 15mm, 부착 위치 통일(뒤표지 우하단 등), 투명 테이프로 보호.
- QR 값 = **URL형** (`https://.../b/0001234`) → 학생이 그냥 폰 기본 카메라로 찍으면 **책 소개·한줄평 페이지**로 이동. 독서 유도 장치가 공짜로 딸려옵니다.
- 학생 QR = **불투명 ID만.** URL 금지 (학생증이 공개 페이지로 연결되면 안 됩니다).
- 5,000장이면 **라벨 프린터(브라더 QL 등, 10만원대)가 A4 라벨지보다 압도적으로 편합니다.** 예산이 없다면 A4 라벨지 + "라벨 인쇄 대기열" 기능(한 페이지 분량 차면 일괄 출력).

### 웹캠 주의

- **고정초점 저가 웹캠은 바코드를 못 읽습니다.** 근접 초점이 안 잡혀요. **오토포커스 웹캠**(로지텍 C920급) 필수.
- 책 표지 코팅 반사 → 조명 각도 조절.
- **ISBN 연속 스캔은 웹캠보다 폰이 훨씬 낫습니다.** 폰으로 스캔 → PC에서 편집, 하이브리드 권장.
- 디코더: `BarcodeDetector` API (Chrome·Edge·Android 네이티브, 매우 빠름) + **iOS Safari 미지원이므로 ZXing-js 폴백 필수.** QR과 ISBN(EAN-13)을 같은 코드로 처리.

## 8. 확장 탈출구

`History_*` 합계가 **5만 행**을 넘으면:

1. 지난 연도 History를 **별도 스프레드시트 파일**로 이전
2. `Stats`는 이미 야간 배치 캐시이므로 사용자 경험에는 영향 없음
3. 그래도 느려지면 → **BigQuery 무료 티어** 또는 Firestore로 History만 이전

`Books`·`Students`·`Loans`는 구조상 커지지 않으므로 **운영 성능은 계속 유지됩니다.** 이 설계의 핵심 이점입니다.
