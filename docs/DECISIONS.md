## ADR-001 · 프론트엔드를 Apps Script 밖에 둔다

**결정** — GitHub Pages(React PWA) + GAS(JSON API) + Sheets(DB)

**왜** — GAS의 `HtmlService` 샌드박스가 `getUserMedia`를 차단합니다. Google이 2026년 6월 공식 문서에 **의도된 제약**으로 못 박았고, 해결책으로 "외부 도메인 호스팅"을 직접 제시했습니다. QR·ISBN 스캔이 이 프로젝트의 핵심이므로 우회로가 없습니다.

**기각** — GAS 웹앱 단독: 2026년 2월부터 카메라가 죽었습니다. 인터넷의 "GAS로 QR 스캐너" 튜토리얼은 전부 그 이전 자료라 무효입니다.

**검증** — 2026-07-13 스파이크. iPhone(iOS 18.7) **설치형 PWA**에서 카메라 + QR/EAN-13 **연속 48건** 성공. `standalone: true` 확인.

**대가** — 인증을 직접 구현해야 합니다 (`Session.getActiveUser()` 공짜 로그인 상실).

## ADR-002 · 도서ID = 행 번호 〔🔄 2026-07-13 시트 계층에서 폐기 — ADR-016〕

> **개정**: MVP 패키지 채택으로 시트 DB는 행-독립 `copy_id`를 사용한다.
> 얻은 것: **정렬 내성** (이 ADR의 최대 취약점 "정렬 한 번 = 라벨 5,000장 무효"가 사라짐).
> 잃은 것: O(1) 조회 → 요청 스코프 캐시(PATCH_SPEC P2)로 보완.
> 웹앱의 카탈로그 미러(IndexedDB 검색)는 이 개정과 무관하게 유지.
> 아래 원문은 기록 보존용.

**결정** — `행 번호 = 도서ID + 1`. 순차 발번. 정렬·삽입·삭제 영구 금지.

**왜** — 장서 5,000권+. 스캔할 때마다 전체를 훑으면 느립니다.
`getRange(도서ID + 1, ...)` 한 번이면 **O(1)** 입니다. 5,000행을 뒤지는 것과 비교가 안 됩니다.

**기각**
- 시트 전체 스캔 → 5,000행 × 하루 300건. GAS 할당량과 응답속도 둘 다 무너집니다.
- `MATCH`로 찾기 → O(n). 나을 게 없습니다.

**대가** — 🔴 **정렬 한 번에 라벨 5,000장이 무효화됩니다.** 되돌릴 방법이 없습니다.

**완화** — 경고문으로는 부족합니다(반드시 누군가 무시합니다). **사서는 DB 시트를 편집하지 않습니다** (ADR-006). 시트 보호를 걸어 **물리적으로 차단**합니다.

## ADR-003 · ISBN을 관리 ID로 쓰지 않는다

> 🚫 **가장 자주 다시 제안되는 대안입니다. 반드시 읽으세요.**

**결정** — 자체 등록번호 + 자체 QR. ISBN은 **등록 시 서지 자동입력에만** 씁니다.

**왜**
1. **ISBN은 "한 권"이 아니라 "책의 종류"입니다.** 『아몬드』 3권의 ISBN이 전부 같습니다.
   → 누가 어느 권을 빌렸는지, 어느 권을 폐기할지 **특정 불가**.
   → 5,000권 규모에 복본은 반드시 존재합니다. 인기 도서는 여러 권 삽니다.
   → **세상의 모든 도서관이 ISBN이 인쇄돼 있는데도 자기 라벨을 붙이는 이유입니다.**
2. **10~20%는 ISBN이 없습니다** — 기증본·구간·전집·원서. 어차피 별도 체계가 필요 → 시스템이 둘이 됩니다.
3. **1D 바코드(EAN-13)는 폰·웹캠으로 읽기 어렵습니다.** QR은 오류정정 30%, 각도 무관, 초점에 관대.
   전용 스캐너가 없는 이 프로젝트에서 **QR이 압도적으로 유리**합니다.
4. 부가기호(ISBN 옆 5자리) 오독, 위치 제각각, 비닐·라벨에 가려짐.
5. ISBN은 숫자일 뿐 → **URL 트릭 상실** (ADR-004).

**대가** — 라벨 부착 노동. 하지만 **청구기호 라벨은 어차피 붙여야 합니다**(서가 배열). 같은 라벨에 QR을 인쇄하면 **추가 비용 0**입니다.

## ADR-004 · 책 QR = URL, 학생 QR = 불투명 ID

**결정** — 책: `https://lib.○○.kr/b/0005001` · 학생: `S:0000421` (URL 아님)

**왜** — 학생이 **폰 기본 카메라**로 책 QR을 찍으면 책 소개 페이지가 열립니다.
앱 설치 불필요, 카메라 권한 불필요. **셀프 대출과 독서 유도가 공짜로 딸려옵니다.**
학생 QR을 URL로 만들면 **학생증이 공개 페이지로 연결**됩니다 — 절대 안 됩니다.

**대가** — 🔴 **도메인이 라벨에 영구 인쇄됩니다.** `*.github.io`로 찍으면 호스팅을 옮기는 순간 전체 QR이 죽습니다.
→ **짧은 자체 도메인 선구매 필수.** 라벨 인쇄 전에.

**주의** — QR은 **신원 표시일 뿐 인증이 아닙니다** (사진 찍으면 복제됨). 학생증 QR로 로그인시키지 마세요.

## ADR-005 · 대장 = 이벤트 원장 (대출 +1, 반납 −1) 〔🔄 2026-07-13 개정 — ADR-016〕

> **개정**: LOANS는 상태 테이블(returned_at UPDATE)로 간다. 이벤트 이력은
> `15_AUDIT_LOG`(before/after JSON)가 담당하고, GAS 사망 시 수기 경로는
> `20_MANUAL_ENTRY` 시트(PATCH_SPEC P3)가 대신한다 — 흡수 시 기존 멱등 체계가 중복을 막는다.
> "+1/−1 자가검증"은 잃지만 무결성 점검(`apiRunIntegrityCheck`)이 그 역할을 한다.
> 아래 원문은 기록 보존용.

**결정** — 대출/반납을 상태가 아니라 **사건**으로 기록. append-only. 수정·삭제 금지.

**왜**
- **책별 합계 = 1 → 대출중. 0 → 서가.** QUERY/SUMIF 한 줄로 현황이 나옵니다.
- **앱 없이 시트만으로 도서관이 굴러갑니다.** 사서가 한 줄 추가하면 그게 대출입니다.
- **오류가 저절로 드러납니다.** 합계 2 = 중복 대출, −1 = 대출 없이 반납. 별도 검증 로직이 필요 없습니다.
- 감사 추적이 공짜입니다.

**기각**
- `Loans`(활성) + `History`(이력) 2개 테이블 → **흡수·동기화 로직**이 필요하고, 실패하면 유령 대출이 남습니다.
- 상태 컬럼만 두기 → 이력이 사라집니다.

**대가** — 대장이 커집니다 → **연도별 분할**(ADR-008).

**수정 방법** — 지우지 않고 **반대 줄을 추가**합니다 (`취소` 구분). 회계 장부와 같습니다.

## ADR-006 · 사람이 쓰는 시트와 앱이 쓰는 시트를 분리

**결정** — 사서는 **`시작` 시트만** 봅니다. DB 시트는 숨김 + 보호.

**왜** — 정규화된 테이블에 사람이 직접 손을 넣는 건 불가능합니다.
수기 대출 한 건 = **필드 9개 · 시트 3개 · 날짜 계산 · 행 번호 암산.** 하루 100건이면 죽습니다.
그리고 "재동기화 버튼을 누르면 된다"는 소용없습니다 — **앱이 죽었으면 그 버튼도 못 누릅니다.**

**부수 효과 (이게 더 큽니다)** — 사서가 `Books`를 편집할 이유가 사라지므로 **보호를 걸 수 있습니다** → ADR-002의 정렬 사고가 **구조적으로 불가능**해집니다.

**시트 순서**
```
1  시작      사서가 보는 유일한 화면 (콘솔 — 표가 아님)
2  사용법
3~ DB        평소 숨김. 앱 전용
```

## ADR-007 · 도서번호는 숫자, 표시만 6자리

**결정** — 셀 값 = `500` (숫자) · 셀 서식 = `000000` → 화면엔 `000500`

**왜** — 🔴 **Google Sheets는 텍스트 `000500`을 숫자 `500`으로 자동 변환합니다.** 앞의 0이 날아갑니다.
라벨의 `000500`과 시트의 `500`이 매칭 안 되고, `MATCH`·`SUMIF`에서 타입이 어긋납니다.

**결과** — 사서가 `000500`을 치든 `500`을 치든 **같은 결과**. 행 번호 = `501`로 직관적. 절대 안 깨집니다.

## ADR-008 · 파일은 나누지 않는다 (시간만 나눈다)

**결정** — 운영 파일 1개. 지난 연도 대장만 별도 보관 파일로.

**왜** — 파일을 나누면 수식이 **`IMPORTRANGE`** 를 써야 합니다.
- 캐시 기반 → **몇 초~수십 초 지연**. `시작` 시트의 **즉시 미리보기가 죽습니다** — 그게 이 설계의 심장인데요.
- `#REF!` 빈발, 수동 액세스 허용 필요.
- Apps Script가 대출 한 건에 `openById()` × 4 → **1~2초 추가 + 할당량 소모**.

**시트 수는 성능과 무관합니다.** 문제는 **행 수와 수식 수**입니다.

```
도서관        시작 · 사용법 · 대장_2026 · Books · Students · 연체 · 설정 · 재계산
도서관_보관    대장_2025 · 대장_2024 …   ← 통계 배치만 읽음. 수식 0개
```

보관 파일은 **아무 수식도 참조하지 않으므로** IMPORTRANGE 문제가 없습니다. 그래서 이것만 나눠도 안전합니다.

## ADR-009 · 파생값은 앱이 값으로 유지 (수식 아님)

> 🔴 **v3에서 발견된 결함의 수정입니다.**

**결정** — `Books.상태 · 대출자 · 반납예정`을 **앱이 값으로** 씁니다. 연체 = Books 필터 (수식 0개).

**왜** — 수식으로 대장을 훑으면 **각 행마다 그 아래 전체를 검사** → **O(n²)**.

| 대장 | 연산 | |
|---|---|---|
| 25행 (데모) | 600 | 즉시 ✅ |
| 10,000행 (1년) | **1억** | 수십 초 🟡 |
| 50,000행 (5년) | **25억** | **시트가 안 열림** 🔴 |

**데모에선 안 보입니다. 1년이면 죽습니다.**

**비상 경로** — `재계산` 시트 (평소엔 **비어 있음**, 수식 0개).
앱이 죽었을 때만 켜서 대장에서 통째로 재계산합니다. 느리지만 작동합니다.

**원칙** — 앱이 죽는 건 드뭅니다. 그때 30초 기다리는 건 괜찮습니다.
**평상시에 매번 30초는 못 견딥니다.**

## ADR-010 · 셀프 대출 O / 셀프 반납 X

> 직관과 반대입니다.

**결정** — 학생은 본인 폰으로 **대출**할 수 있습니다. **반납은 불가.**

**왜** — 대출은 책임을 만들고, 반납은 책임을 **지웁니다. 지우는 쪽이 위험합니다.**
- 셀프 반납 허용 → 학생이 집에서 "반납"을 누르고 책을 계속 가집니다. 시스템은 서가에 있다고 믿습니다.
  **장서점검 때까지 몇 달간 아무도 모릅니다.**
- 셀프 대출 악용 → 당사자가 바로 항의합니다. 즉시 발각, 기록에 기기·시각이 남습니다.

**그리고 셀프 반납은 편의성 이득이 0입니다** — 책을 돌려주려면 어차피 도서관에 와야 합니다.
반면 셀프 대출은 사서가 자리를 비웠거나 줄이 길 때 실질 이득이 큽니다.
(셀프 대출의 진짜 대안은 "올바른 대출"이 아니라 **"그냥 들고 나감"** 입니다.)

**결정** — 도우미 학생에게 계정을 만들지 않습니다. **기기 등록**(6자리 코드 → 토큰).

도우미가 반 친구 독서기록을 뒤지는 걸 막는 선입니다.

## ADR-012 · 삭제하지 않는다

**결정** — 책·학생·대출 모두 **행을 지우지 않습니다.** `상태`만 바꿉니다.

**왜**
- 행 삭제 → **그 아래 모든 ID가 밀립니다** (ADR-002). 라벨·QR이 전부 엉뚱한 대상을 가리킵니다.
- 과거 대출 기록이 **미아**가 됩니다. 대장의 `학생번호 42`가 누구인지 알 수 없게 됩니다.

**결과** — 폐기해도 학생 독서기록에 "『○○○』 (폐기됨)"으로 정상 표시됩니다. 통계도 안 깨집니다.
**번호는 재사용하지 않습니다.**

**예외** — 학생 **정보 수정**은 자유입니다 (이름·학년·반). **`학생번호`만** 못 고칩니다.

## ADR-013 · 자료유형을 지금 넣는다 〔🔄 2026-07-13 강화 — 정책 매트릭스로 흡수〕

> **개정**: `자료유형+대출가능+대출일수` 컬럼 대신 MVP 패키지의
> `13_POLICIES` 매트릭스(회원유형 × 자료유형)로 표현한다. 참고도서 = 한도 0 정책 한 줄.
> 원래 취지("나중에 넣으면 5,000권 재분류")는 그대로 유효 — material_type 코드를 지금 확정한다(P1-b).

**결정** — `Books.자료유형` + `대출가능` + `대출일수`(개별 override)

**왜** — 없으면 **국어대사전이 2주간 대출됩니다.** 그 반 아이들 전체가 못 씁니다.

**타이밍이 전부입니다** — 5,000권을 다 등록한 뒤에 컬럼을 추가하면 **5,000권을 손으로 다시 분류**해야 합니다.
지금 넣는 컬럼은 **공짜**, 나중에 넣는 컬럼은 **재분류 노동**입니다.

같은 이유로 지금 넣은 것: `세트명/순번`(전집) · `단가`(예산 보고) · `기증자` · `난이도` · `권장학년`

## ADR-014 · 무료 Gmail로 충분하다

**결정** — 개인 Gmail 계정. Workspace 불필요.

**왜** — 인가 학교가 아니라 무상 Workspace(Education)를 못 받습니다.
GAS 일일 실행시간은 무료 계정 **90분**인데, 실제 계산은:

| | |
|---|---|
| 대출·반납 200~300건 | 7~10분 |
| **검색** | **0초** ← 카탈로그 미러로 브라우저에서 처리 |
| 조회·야간 통계 | 10~15분 |
| **합계** | **20~30분 (90분의 1/3)** |

**5,000권 초기 등록도 문제없습니다.** 사람이 하루에 등록 가능한 건 200~300권이 한계 → 하루 15분.
**사람 손이 서버보다 느립니다.**

**진짜 위험** — 버그 폭주(무한 루프 → 90분을 몇 분 만에 소진, **경고 없이 전체 정지**)와

## ADR-015 · AppSheet는 보험, 대체재가 아니다

**결정** — 지금은 만들지 않습니다. Phase 5 이후 검토.

**왜 검토 가치가 있나** — GAS·Pages와 **완전히 독립**적으로 Sheets를 읽고 씁니다.
GAS 할당량이 터져도, React에 버그가 나도 멀쩡합니다. 그리고 **노코드라 개발자가 떠난 뒤에도 후임이 고칠 수 있습니다.**

- 🔴 **AppSheet의 "삭제"가 행을 지웁니다** → ADR-002 파괴. 반드시 비활성화해야 합니다.

# 방법론 — 어떻게 여기까지 왔나

## 1. 가장 불확실한 것을 가장 먼저

기능 순서대로 만들면 **가장 위험한 걸 가장 나중에** 만납니다. 그때는 매몰비용이 큽니다.

이 프로젝트의 최대 위험은 "iOS 설치형 PWA에서 카메라가 켜지는가"였습니다.
**반나절짜리 스파이크**로 먼저 죽였습니다. 실패했다면 아키텍처를 통째로 갈아엎어야 했습니다.

## 2. 문서보다 더미 데이터

**설계는 종이 위에서 항상 맞아 보입니다.** 실제로 만들어보면 다릅니다.

더미 시트를 만들어서 발견한 것들 — **전부 문서로만 봤으면 놓쳤습니다:**

| 발견 | 증상 |
|---|---|
| 중복 대출이 **연체를 은폐** | 박지호의 6일 연체가 목록에서 사라짐 |
| 재대출된 책이 **증발** | 『채식주의자』『아몬드』가 미반납 목록에서 빠짐 |
| `설정` 행이 밀림 | **대출일수(14) 대신 대출권수(3)를 읽음.** 에러 없음. 그냥 틀린 답 |
| `Books` 컬럼 밀림 | 대출일수 칸에 난이도("상") → `#VALUE!` |
| **O(n²) 수식** | 데모는 즉시. 1년 뒤 시트가 안 열림 |

> **수식이 계산된다는 것과 답이 맞다는 것은 다릅니다.**
> 오류 0개짜리 워크북이 조용히 틀린 답을 냅니다. **값을 직접 검증하세요.**

## 3. 실수를 일부러 심는다

데모에 **없는 번호 · 대출 없이 반납 · 중복 대출**을 심어두고, 시스템이 잡아내는지 봤습니다.
"정상 동작"만 테스트하면 아무것도 검증되지 않습니다.

## 4. 경고문 대신 물리적 차단

"정렬하지 마세요"는 **반드시 누군가 무시합니다.**
→ 사서가 DB 시트를 볼 이유 자체를 없애고, 보호를 걸었습니다 (ADR-006).

## 5. 되돌릴 수 없는 것을 먼저

**도메인 · 등록번호 체계 · 자료유형 컬럼** — 라벨 5,000장을 붙인 뒤엔 못 바꿉니다.
코드는 언제든 고칩니다. **물리적으로 고착되는 것부터** 확정합니다.

## 6. 사람 손이 서버보다 느리다

할당량·성능을 걱정하기 전에 **인간의 처리 속도**를 계산했습니다.
5,000권 등록 = 85시간 노동. 하루 300권이 한계 → GAS는 하루 15분.
**병목이 어디인지 알면 최적화할 곳도 알게 됩니다.**

## ADR-016 · 외부 MVP 패키지 채택 — ADR-002·005의 시트 계층 개정

**결정** — 사용자가 가져온 정규화 RDB형 GAS 패키지(TITLES/COPIES/LOANS 19시트)를 **기반으로 채택**하고 학교화 패치를 적용한다. 재작성하지 않는다.

**왜** — 검증된 2,179줄(실제 시트 보호 + 소유자 예외 처리, request_id 멱등 + 보상 트랜잭션, 예약 대기열 자동 배정, before/after 감사 로그)을 버리는 것은 낭비. 정책 매트릭스(회원유형×자료유형)는 우리 자료유형 컬럼 설계보다 우아함. 우선순위 3(웹앱 확장·RDB 이전)에 직행 구조.

**개정되는 기존 결정**
- **ADR-002 (도서ID=행 번호)** → 시트 계층에서 폐기. ID는 행과 무관한 `copy_id`/`barcode`. O(1)을 잃는 대신 **정렬 내성**을 얻음 (정렬해도 시스템이 안 깨짐 — 기존 설계의 최대 취약점 제거). 성능은 요청 스코프 캐시로 보완.
- **ADR-005 (append-only 대장)** → 시트 계층에서 폐기. LOANS는 상태 변이 테이블. 자가검증(+1/−1)을 잃는 대신 AUDIT_LOG(before/after JSON)가 이벤트 이력 담당. **GAS 사망 시 수기 경로는 Phase B 수기입력 시트로 복원 예정.**
- ADR-003(ISBN≠관리ID)·004(QR=URL)·007(숫자+서식→**텍스트 서식으로 변형 유지**)·010(셀프반납 금지)·013(자료유형)은 **유지**되며 패치로 정합화됨.

**대가** — 사람이 DB 시트를 직접 읽기 어려움(ID 관계형). 운영센터 비상 조회(순수 수식)로 부분 보완, 완전한 수기 경로는 Phase B.

## ADR-017 · 연체 페널티 = 대출 정지 (연체료 아님)

**결정** — `OVERDUE_SUSPEND_MULTIPLIER`(연체 1일=정지 1일). FINES는 분실 변상(REPLACEMENT) 전용. 미변상 시 대출 차단. 사서 재량 해제 체크박스 제공.

**왜** — 학교는 학생에게 돈을 받지 않는다. 원본 패키지의 연체료 모델은 공공도서관용.

## ADR-016 · 사용자 제작 MVP 패키지를 시트·사이드바 계층의 기반으로 채택 (2026-07-13)

**결정** — 사용자가 가져온 정규화 RDB형 패키지(19시트 · Code.gs 2,179줄 · 통합 사이드바)를 채택하고, `PATCH_SPEC.md`의 7개 패치로 학교화한다. 웹앱(Pages + GAS Web App)은 기존 계획 그대로 — 패키지 README 스스로 "다중 사용자는 별도 Web App 필요, 미포함"이라 명시했고 그 반쪽이 우리 로드맵이다.

**왜**
1. **검증된 자산 재사용** — 실제 시트 보호(소유자 강등 차단·실패 시 원복), `request_id` 멱등 + 보상 트랜잭션, 예약 대기열(반납 시 자동 배정·수령기한·타회원 배정분 차단), before/after 감사 로그. 재작성은 주 단위, 패치는 일 단위.
2. **정렬 내성** — ID가 행과 무관해 우리 설계의 최대 취약점(정렬 사고)이 구조적으로 없다.
3. **우선순위 3(웹앱 확장·RDB 이전) 직행** — ID·관계 중심 테이블이라 이전 비용 최소.
4. 정책 매트릭스·CODEBOOK이 우리 자료유형·설정 설계의 상위호환.

**기각한 대안** — v4 플랫 스키마 전면 유지. 비상 자립(수기 한 줄 대출)은 최강이지만, 구현된 2,179줄을 버리고 예약·보호·감사를 재구현해야 하며 RDB 이전 비용이 크다. 비상 자립은 P3(수기입력)+P6(불사 미리보기)로 "충분히" 회수 가능.

**대가와 완화**
- GAS 사망 시 수기 경로 약화 → P3 `20_MANUAL_ENTRY` (비보호 append 시트 + 멱등 흡수)
- 요청당 전체읽기 8~10회 → P2 요청 캐시 + LOANS 연도 아카이브
- 공공도서관 도메인(연체료·주소 수집·학년 없음) → P1 학교화 (연체료→대출정지 전환 포함)
- "상태값은 한글" 원칙 폐기 → CODEBOOK `label_ko/en` 매핑으로 대체 (ADR-017과 정합)

**영향받은 ADR** — 002(시트 계층 폐기) · 005(개정) · 013(정책 매트릭스로 흡수·강화) · 008(아카이브 원칙 유지, LOANS_YYYY로 계승) · 009(패키지가 이미 준수 — 운영센터 수식 0개) · 003/004(유지 — barcode 체계를 P5로 정합)

## ADR-017 · 한/영 다국어: 데이터는 코드, 표시는 사전 (2026-07-13)

**결정** — 데이터 계층은 언어중립 코드(CODEBOOK `label_ko`/`label_en`), 사이드바는 사용자별 토글(`UserProperties`), 시트 콘솔은 `01_운영센터`+`01_Console_EN` 2장.

**왜** — 실제 조작은 전부 사이드바에서 일어나므로 사용자별 전환이 정확히 필요한 곳에 있다. 시트는 공유 화면이라 사용자별 언어가 불가능 → 시트 2장이 전역 토글보다 실용적(각자 자기 탭). 서버는 이미 오류 코드를 반환하므로 클라이언트 매핑만으로 번역 완성.

**정정 기록** — 이전에 "내부 코드가 영어라 사람이 읽기 힘들다"고 지적했으나, 다국어 관점에서는 영어 코드+라벨 매핑이 정답이었다. 구 원칙("시트 상태값은 한글로 — 사서가 직접 고쳐도 앱이 알아듣게")은 폐기: 사서는 이제 DB 시트를 직접 고치지 않는다(보호 + 수기입력 시트 경로).

**대가** — 문자열 ~250개 관리 비용. 미결: 영어 사용자 범위(교직원만인지 학생 포함인지)에 따라 웹앱 i18n 우선순위 결정.
