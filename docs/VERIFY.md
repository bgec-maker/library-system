# VERIFY.md — 통합 웹앱(webapp/) 검증 절차

이 문서는 실제 기기·실제 GAS 배포로만 확인 가능한 것을 검증하는 절차입니다. 이 환경(코드 작성 세션)에는
GAS 런타임도, 실제 아이폰도, 화면이 있는 브라우저도 없습니다. 여기서는 **정적 검증(lint·타입체크·빌드)과
헤드리스 브라우저(Playwright, 임시 설치 후 제거) 스모크 테스트까지만** 수행했고, 아래 절차는 **사용자가
실제 폰·데스크톱 브라우저로 수행**해야 합니다 — school-patch-v1의 9단계 검증 프로토콜과 같은 성격입니다.

## 이미 확인된 것 (이 세션에서, 자동으로)

**확인 못 한 것 — 아래 절차로 사용자가 확인해야 함**: 실제 카메라·ZXing 디코드, iOS Safari 설치형 PWA 동작,
실제 대출·반납/등록이 시트에 실제로 반영되는지, 창 리사이즈 8방향 핸들의 실제 사용감, 모바일 탭 전환 애니메이션,
좌우 스냅, 최대 6개 창 제한의 실제 UX.

**⚠️ 알려진 스코프 축소 — CLAUDE.md 규칙 7 부분 미충족**: 규칙 7은 "ZXing 로컬 번들 + Worker + 크롭 +
10fps 스로틀"을 요구합니다. `camera.ts`는 이 중 **로컬 번들·크롭·10fps 스로틀 셋만** 구현했고 **Web Worker
오프로딩은 넣지 않았습니다**(메인 스레드에서 디코드) — register.html 때와 동일한 트레이드오프이며
`school-patch-v1/PATCH_NOTES_MOBILE_REG.md`에도 같은 내용이 기록되어 있습니다. 2절의 "20권 연속 스캔"에서
발열·버벅임이 실제로 문제라면, 다음 개선 항목은 ZXing 디코드를 Worker로 옮기는 것입니다(현재는 크롭으로 처리
픽셀을 줄여 완화만 함).

## 0. 사전 준비

1. **GAS 백엔드는 이미 배포되어 있다고 가정합니다** (`school-patch-v1/Code.gs`는 이 작업에서 손대지 않았습니다).
   Web App URL, `MOBILE_REG_TOKEN`, 작업자 이름을 준비하세요 — `school-patch-v1/PATCH_NOTES_MOBILE_REG.md`의
   "사전 준비 (설치 순서)" 참고.
2. `.github/workflows/pages.yml`이 `main` 브랜치 push마다 `webapp/`을 빌드해 Pages의 `/app/` 경로에,
   `spike/`를 그대로 사이트 루트에 병행 배포합니다. 즉:
   - 기존 스파이크·등록 도구: `https://bgec-maker.github.io/library-system/` (그대로 살아있음)
   - 새 통합 웹앱: `https://bgec-maker.github.io/library-system/app/` ← 이번에 새로 생김
3. push 후 GitHub Actions "Deploy to GitHub Pages" 워크플로가 성공했는지 Actions 탭에서 확인하세요.
   `webapp 뷰 경계 린트` 스텝이 실패하면 배포 자체가 안 됩니다(의도된 CI 게이트).

## 1. 데스크톱 브라우저에서 (노트북/PC)

1. `/app/`을 열면 **초기 설정 화면**이 뜹니다 — GAS Web App URL·공유 토큰·이름(작업자)을 입력하고 저장하세요.
   나중에 좌측 도크 맨 아래 ⚙로 다시 열 수 있습니다.
2. **창 관리자**: 좌측 도크 아이콘을 클릭해 "대출·반납"과 "도서 등록"을 각각 열어보세요.
   - [ ] 타이틀바를 드래그하면 창이 움직인다
   - [ ] 모서리를 드래그하면 리사이즈된다 (최소 크기 아래로는 안 줄어든다)
   - [ ] 최소화하면 도크 하단에 아이콘으로 쌓이고, 클릭하면 복원된다
   - [ ] 좌/우 스냅 버튼을 누르면 화면 절반을 차지한다
   - [ ] 창을 닫았다 같은 아이콘을 다시 클릭하면(single 뷰) 새 창이 아니라 **같은 창이 포커스**된다(중복 생성 안 됨)
   - [ ] 창을 7개째 열려고 하면 토스트로 거부된다(최대 6개)
   - [ ] 브라우저를 새로고침해도 창 위치·크기가 그대로 복원된다
3. **스캐너 도크 위젯**(우하단): 브라우저가 카메라 권한을 요청합니다 — 허용하세요.
   - [ ] 웹캠 프리뷰가 뜬다 (macOS/Windows Chrome이면 BarcodeDetector 네이티브, 그 외엔 ZXing)
   - [ ] 접기 버튼을 눌러도 카메라 스트림 자체는 안 꺼진다(다시 펼치면 바로 프리뷰가 보임 — 껐다 켜는 지연 없음)
   - [ ] 창을 전부 닫아도 스캐너 위젯은 그대로 살아있다("카메라는 창이 아니다")
4. **스캔 라우팅**: "대출·반납"과 "도서 등록" 창을 동시에 띄우고 번갈아 클릭(포커스 전환)하면서 책 바코드를 스캔:
   - [ ] 포커스된 창으로 스캔이 간다 (다른 창은 반응 없음)
   - [ ] 핀 아이콘을 누르면 포커스를 옮겨도 핀 고정된 창으로 계속 스캔이 간다
   - [ ] 두 창을 모두 닫은 상태에서 스캔하면 "대출·반납" 창이 자동으로 열린다

## 2. 아이폰 — 설치형 PWA (진짜 시험)

**브라우저 탭에서 되는 건 이미 압니다. 설치형에서 되는지가 관건입니다** (스파이크 때와 같은 원칙).

1. Safari로 `https://bgec-maker.github.io/library-system/app/` 접속 → **공유 → 홈 화면에 추가** → 그 아이콘으로 다시 열기.
2. 초기 설정 화면에서 URL·토큰·이름 입력.
3. 하단 탭바가 보이고 **첫 탭이 "스캔"(=대출·반납 뷰)** 인지 확인.
4. **도서 등록 탭**으로 이동해 ISBN 스캔 20권 연속 시도:
   - [ ] 연속으로 인식되는지 (한 번 읽고 멈추면 실패)
   - [ ] 발열·크래시 없는지
   - [ ] 부가기호(ISBN 옆 5자리 바코드)를 스캔했을 때 무시되고 계속 대기 상태인지
5. "더보기" 탭에서 "장서 점검"(inventory, 스텁이지만 scan:focus)에 들어가서 스캔하면 카메라가 켜지는지(탭 전환 시
   카메라 on/off가 배터리를 고려해 전환되는지) 확인.
6. 뒤로가기 제스처: 도서 상세 같은 push 화면에서 뒤로가기 → 이전 화면으로. 탭 루트에서 뒤로가기 → 앱이 죽지 않고
   탭이 유지되는지 확인(뒤로가기가 Safari를 벗어나거나 PWA를 종료하면 안 됨).

## 3. 🔴 스모크 버그 진단 — "스캔·저장까지 갔는데 시트에 안 보임"

register.html에서 미해결로 남아있던 버그입니다. 새 앱(`views/register`)도 **같은 doPost**를 쓰므로 여기서
재현될 수 있습니다. 이번에 진단 전용 UI를 register 뷰에 넣었으니 아래 순서로 원인을 좁히세요.

1. "도서 등록" 탭(또는 창) 헤더의 **"진단 로그 보기"**를 눌러 패널을 엽니다.
2. 평소처럼 ISBN 스캔 → 확인 폼 → 저장을 진행합니다.
3. 저장 직후 화면에 **등록번호와 함께 "요청 ID: xxx"**가 표시됩니다. 이 값을 적어두세요(또는 "진단 로그 복사"로
   전체를 클립보드에 복사).
4. 진단 로그 패널에서 방금 요청(`registerByIsbn`)의 줄을 찾아 다음을 확인:
   - **outcome이 'ok'가 아니면** → 프론트가 실패를 감지한 것입니다. `resultCode`/`resultMessage`를 그대로
     스크린샷하거나 복사해 알려주세요 (프론트 문제일 가능성 — 예: `NETWORK_ERROR`/`CLIENT_TIMEOUT`이면
     요청이 서버에 아예 안 닿았거나 응답을 못 받은 것).
   - **outcome이 'ok'인데 시트에 안 보이면** → 요청은 서버까지 갔고 서버도 성공을 응답한 것입니다. 이제
     서버 쪽을 봐야 합니다:
     a. Apps Script 편집기 → **실행 기록(Executions)**에서 저장 시각 근처의 `doPost` 실행을 찾아 로그를 확인하세요.
     b. 스프레드시트에서 **`18_SYS_OPERATIONS`** 시트(기본은 숨김 — 메뉴에서 잠깐 보이게 하거나 시트 목록에서
        직접 선택)를 열어 4단계에서 적어둔 requestId로 검색(Ctrl+F)하세요.
        - 행이 아예 없다 → 요청이 **다른 GAS 배포/다른 시트**로 갔을 가능성이 큽니다. 초기 설정에 입력한
          Web App URL이 지금 보고 있는 스프레드시트에 바인딩된 배포가 맞는지 다시 확인하세요(여러 번 배포하면서
          URL이 여러 개 생겼을 수 있습니다 — 가장 최근 활성 배포 URL인지 확인).
        - `status_code`가 `COMPLETED`다 → 서버는 정상적으로 다 처리했다고 기록했습니다. 이 경우 `08_COPIES`
          시트(역시 기본 숨김)에서 응답에 찍힌 barcode로 검색하세요. **여기서도 없으면 진짜 서버 버그**입니다
          — `registerByIsbn`/`registerTitle_`/`registerCopy_` 쪽을 봐야 합니다.
        - `status_code`가 `STARTED`에 멈춰 있다 → 실행이 중간에 죽은 것입니다(타임아웃 등). 무결성 점검
          메뉴를 실행해보세요.
        - `status_code`가 `FAILED`다 → `error_message` 열에 실패 사유가 있습니다.
   - 화면엔 보이는데 사용자가 못 찾았을 가능성도 있습니다 — `03_TITLES`/`08_COPIES`는 **기본적으로 숨김
     시트**입니다(사서가 실수로 정렬·수정하지 못하게 하는 설계, `docs/DECISIONS.md` ADR-006). 운영센터
     대시보드 숫자도 "새로고침" 메뉴를 눌러야 갱신됩니다 — 자동 갱신 아님.
5. 위 단서(진단 로그 outcome, requestId, 18_SYS_OPERATIONS 상태, 08_COPIES 존재 여부)를 모아서 알려주시면
   프론트/서버 어느 쪽이 원인인지 바로 좁혀집니다.

## 4. 대출·반납 뷰 — 알려진 백엔드 공백

`views/loan-return`은 `checkout`/`return` 액션을 실제로 호출하도록 만들었지만, **`school-patch-v1/Code.gs`의
`doPost`는 아직 `lookupIsbn`/`registerByIsbn`만 라우팅합니다** (이번 작업은 Code.gs를 수정하지 않는 게
조건이었습니다). 그래서 지금 대출·반납을 실행하면 화면에 "서버에 checkout/return 액션이 아직 없습니다"라는
오류가 그대로 뜨는 게 **정상**입니다 — 가짜 성공이 아닙니다.

실제로 동작하게 하려면 `doPost`의 action 분기에 아래 두 줄을 추가하면 됩니다(기존 `apiCheckout`/`apiReturn`과
동일한 패턴, `executeWrite_`를 그대로 재사용):

```js
if (action === 'checkout') return apiCheckout(payload);
if (action === 'return') return apiReturn(payload);
```

프론트가 보내는 payload는 이미 `checkout_`/`return_`의 실제 시그니처와 일치합니다
(`{memberKey, copyKey, note, requestId, token, action}` / `{copyKey, note, requestId, token, action}`).
이 변경은 Code.gs를 직접 여는 작업이라 이번 세션 범위 밖에 뒀습니다 — 원하시면 다음 라운드에서 진행하겠습니다.

## 5. 기존 배포 철수 계획 — 지금은 1~2단계까지만

- ✅ 1단계(병행 배포): `/app/`에 새 앱, 기존 스파이크·`register.html`은 그대로.
- ⏳ 2단계(검증): 이 문서의 절차로 사용자가 실기기에서 확인.
- ⛔ 3단계(리다이렉트 스텁으로 교체)는 **아직 하지 않았습니다** — 지시대로 승인 대기 중입니다. 2단계가
  끝나 확실해지면 알려주세요.
