# TASK_MOBILE_REG.md — 폰 ISBN 등록 도구 (0단계 착수)

> 우선순위: **최상.** 이게 없으면 5,000권 등록을 시작할 수 없다 (권당 2~3분 수동 → 30초).
> 이 도구는 웹앱 트랙(Pages + GAS Web App)의 **첫 벽돌**이다 — 스캐너·API 패턴을 이후 QR 대출관리가 그대로 물려받는다.

## 현재 상태 (전제)

- `school-patch-v1`이 실제 시트에 설치 중 (사용자가 9단계 프로토콜 검증 예정) — **이 GAS 프로젝트에 코드를 추가한다**
- 스파이크 배포 완료: `https://bgec-maker.github.io` — 아이폰 설치형 PWA에서 ZXing으로 **EAN-13 연속 판독 검증됨**
- 도메인 미구매 → **QR 라벨 인쇄는 차단** 상태지만 등록은 무관 (등록번호 발급은 라벨과 독립)
- 확인된 사실: 현행 Code.gs에 `doPost`/`doGet` **없음**, `UrlFetchApp` **없음** → 둘 다 이 태스크에서 추가

## 아키텍처

```
[폰 PWA — Pages]                     [GAS Web App — 같은 프로젝트에 추가]
스파이크 스캐너 확장                    doPost(e) JSON 라우터
EAN-13 스캔 → ISBN13 검증     ───▶    action: lookupIsbn | registerByIsbn
확인·보정 폼 → 저장                    ├ lookupIsbn: 알라딘 → (국중 보강) + BookCache
등록번호 크게 표시 → 다음 스캔          └ registerByIsbn: registerTitle_/registerCopy_ 재사용
                                        (executeWrite_ 경유 = requestId 멱등 공짜)
```

- 배포: **"나(소유자)로 실행" + "모든 사용자"** — CORS가 이 조합에서만 열림 (기존 결정)
- 인증: `ScriptProperties`의 공유 토큰 1개 + **기기별 `operator`(작업자 이름)**. 프론트가 최초 1회 토큰·이름을 입력받아 localStorage 저장, 매 요청 동봉. 등록 스프린트용 임시 방편 — 정식 로그인(🟡 미정)과 별개이며, QR 대출관리는 로그인 결정 후
- 🔴 **공개 레포에 토큰·API 키 절대 금지** — 전부 ScriptProperties

## GAS 측 작업

1. `appsscript.json`에 `https://www.googleapis.com/auth/script.external_request` 스코프 추가
   ⚠️ **스코프 변경 = 재승인 필요.** 사용자에게 "다시 승인 화면이 뜨는 게 정상"이라고 안내할 것
2. `doPost(e)`: JSON 파싱 → 토큰 검증(불일치 시 401) → action 분기 → `ContentService` JSON 반환
3. `apiLookupIsbn`:
   - `CacheService`(6h) → `21_BOOK_CACHE` 시트 → 미스 시 **알라딘 ItemLookUp**(서명·저자·출판사·발행년·**페이지수**·표지URL) → 국중은 키 확보 후 보강(선택)
   - 반환 필드는 `registerTitle_` payload와 1:1 매핑
4. `registerByIsbn` action: `executeWrite_('REGISTER_BY_ISBN', payload, cb)` 안에서
   - ISBN으로 TITLES 조회 → 없으면 `registerTitle_` → 있으면 기존 title_id 사용 (기존 `DUPLICATE_ISBN` 가드를 **분기 신호**로 활용)
   - `registerCopy_` × 복본수 → 발급된 `barcode` 배열 반환
5. 응답에 항상 `{ok, data|error:{code,message}}` — 기존 `fail_` 규약 유지

## 프론트 측 작업 (Pages — 스파이크 레포 확장)

- 스캔 루프는 스파이크 코드 재사용. **iOS 제약 그대로 적용**: ZXing 로컬 번들 · Web Worker · 조준 프레임 크롭 · 10fps 스로틀 (장시간 등록 세션 = 발열 주의)
- EAN-13 체크디짓 검증 → `978`/`979` 시작만 ISBN으로 인정 (부가기호 5자리 바코드 오독 차단)
- 흐름: 스캔 → 조회 스피너 → **확인·보정 폼**(서명·저자·출판사·발행년·페이지·복본수. 자동 저장 금지 — 사서 확인 필수) → 저장 → **등록번호를 화면 가득 크게** → [다음 스캔]
- `requestId = crypto.randomUUID()` — 타임아웃 재시도 시 동일 ID 재전송 (중복은 서버 멱등이 흡수)
- 세션 카운터("오늘 37권") + 실패 목록(나중 재시도용)
- `DUPLICATE_ISBN` 응답 → "이미 있는 책입니다. **복본으로 추가할까요?**" 원탭 분기

## 🔴 병렬 다중 사용자 (필수 요구 — 여러 명이 동시에 스캔한다)

1. **작업자 식별**: 모든 쓰기 요청에 `operator` 포함 → 등록 기록에서 **누가 등록했는지 식별 가능**해야 함 (감사 로그 detail 또는 note에 기록. Web App은 소유자 권한 실행이라 `Session`으로는 구분 불가 — payload로 받아야 함)
2. **동일 ISBN 레이스**: 두 폰이 같은 신규 ISBN을 거의 동시에 등록하면 늦은 쪽이 `DUPLICATE_ISBN`을 받는다 (ScriptLock 직렬화). 클라이언트는 이를 **register 단계에서도** 처리해야 함: `DUPLICATE_ISBN` 수신 → 기존 title 조회 → **자동으로 복본 추가 플로우로 폴백** (병렬 작업에선 이 충돌이 일상임 — 사람 A가 『아몬드』 1권, 사람 B가 2권째)
3. **락 대기**: 쓰기는 전부 ScriptLock으로 직렬화됨 → 동시 요청은 줄을 선다. 클라이언트 타임아웃 30초 + **동일 `requestId`로 재시도** (멱등이 중복 흡수)
4. GAS 동시 실행 한도(~30)와 소유자 단일 할당량 공유 — 10명 이하 스프린트면 무리 없음. 폭주 재시도(지수 백오프 없는 무한 재시도) 금지

**추가 수용 기준:**
- [ ] 두 기기가 같은 신규 ISBN을 동시 등록 → 서지 1건 + 복본 2건 (서지 중복 0)
- [ ] 3대가 각 20권 연속 등록 → 등록번호 중복 0, 유실 0
- [ ] 각 소장본 기록에서 operator 식별 가능

## 범위에서 뺀 것 (의도적 — 다시 넣지 말 것)

- **KDC·청구기호 입력** — 라벨 단계(도메인 확보 후)에서 일괄 부여. 지금 넣으면 권당 30초가 90초 된다
- **표지 촬영·AI 서지추출** — ISBN 없는 책(10~20%)은 이 도구 범위 밖. 사이드바 수동 등록으로
- **오프라인 큐** — 등록은 도서관 와이파이에서 함. 재시도+멱등이면 충분

## 🔴 물리 매칭 규칙 (도구 밖 — 사용자 안내 필수)

라벨은 나중에 일괄 인쇄한다. 그 사이 **어느 책이 몇 번인지**를 잃으면 복본 매칭이 불가능해진다.
→ **등록 직후 등록번호를 책 속표지에 연필로 기입** (화면에 크게 띄우는 이유). 이 규칙 없이 300권 등록하면 되돌릴 수 없다.
→ 병렬 작업 시 **사람 = 구역(서가/수레)**. 같은 서가를 두 사람이 만지면 이중 등록 위험. 완료 구역은 표시.

## 사용자가 미리 할 일

1. **알라딘 TTB 키 발급** (즉시) → ScriptProperties `ALADIN_TTB_KEY`
2. 국중 API 키 신청 (리드타임 있음, 후속 보강용)
3. 웹앱 배포 후 URL + 토큰을 프론트 설정에
