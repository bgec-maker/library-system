**운영 주체:** 정식 인가 학교가 **아님**. 예산·인력 부족. 무료 티어로만 구성.

## 2. 🔴 하드 제약 — 어기면 조용히 망가집니다

이 목록이 이 문서에서 가장 중요합니다. 대부분 **에러 없이 실패**합니다.

### ① GAS 안에서 카메라(`getUserMedia`)는 작동하지 않습니다

2026년 2~3월 Google이 Apps Script 샌드박스에서 camera/microphone을 차단했고, **6월에 공식 문서로 확정**했습니다 (버그 아님, 정책).
에러: `Permissions policy violation: camera is not allowed in this document`

> **따라서 프론트엔드는 반드시 GAS 밖에 있어야 합니다.**
> 인터넷의 "GAS로 QR 스캐너 만들기" 튜토리얼은 전부 2026년 2월 이전 자료이므로 무효.
> 이 프로젝트는 QR 스캔 장비가 **폰 카메라 + 웹캠뿐**이므로 우회로가 없습니다.

### ② 스프레드시트를 정렬(Sort)하면 시스템이 파괴됩니다

`도서ID = 행 번호 − 1` 규칙으로 O(1) 조회를 합니다.

- 🚫 **정렬 금지** (Books, Students)
- 🚫 **행 삽입·삭제 금지**
- 🚫 **ID 재사용 금지**

정렬 한 번이면 **5,000장의 라벨이 전부 무효화**되고 복구할 수 없습니다.
→ 시트 보호(Protected range) 걸 것. README 탭 최상단에 경고 명시. 사서에겐 **필터 보기(Filter views)** 안내.

### ③ 책의 "속성"을 ID에 인코딩하지 마세요

난이도·분류·서가 위치 등을 등록번호 구간으로 나누려는 시도(예: `10001~19999 = 하`)는 **거부되었습니다.**
속성은 **바뀌지만 ID는 불변**입니다. 난이도가 바뀌면 라벨·QR을 다시 붙여야 하고 과거 대출 이력이 깨집니다.

→ 속성은 전부 **컬럼**으로. 물리적 표시는 **교체 가능한 컬러 스티커**로.

### ④ 멱등키(`opId`) 없으면 대출이 중복 생성됩니다

오프라인 큐를 만드는 순간 필연입니다. 모바일 타임아웃 → 사용자 재시도 → 중복.
**모든 쓰기 요청에 클라이언트가 UUID를 붙이고, 서버는 처리한 opId를 무시**합니다.

### ⑤ 카메라는 싱글턴입니다

한 기기에서 두 컴포넌트가 각각 `getUserMedia`를 부르면 충돌하거나 실패합니다.
→ `CameraService` **하나만** 스트림을 소유. 창들은 `scanBus`를 구독. **포커스된 창이 스캔을 받습니다.**
→ **`<video>`를 언마운트하지 마세요.** 창 최소화 = offscreen 이동/`visibility:hidden`. 언마운트하면 스트림 재시작(2~3초 손해).

### ⑥ 권한 검증은 반드시 서버(GAS)에서

프론트에서 버튼을 숨기는 건 UI일 뿐 보안이 아닙니다. GitHub Pages 무료 플랜은 **레포가 공개**이므로 더더욱.
→ **프론트에 비밀값을 절대 넣지 마세요.**

### ⑦ 셀프 **반납**은 금지, 셀프 **대출**은 허용

직관과 반대입니다. 대출은 책임을 만들고 반납은 책임을 지웁니다. **지우는 쪽이 위험합니다.**
셀프 반납을 열면 학생이 집에서 "반납"을 누르고 책을 계속 가질 수 있고, 시스템은 서가에 있다고 믿습니다 — 장서점검 때까지 아무도 모릅니다. 게다가 **편의성 이득이 0**입니다 (책을 돌려주려면 어차피 와야 하므로).

### ⑧ 도메인 확정 전에 라벨을 인쇄하지 마세요

QR에 URL이 들어가므로 **도메인이 5,000장 라벨에 영구 인쇄**됩니다.
`*.github.io`로 찍으면 나중에 호스팅을 옮기는 순간 **전체 QR이 죽습니다.**
→ **짧은 자체 도메인 선구매 필수** (예: `lib.○○.kr`). 짧을수록 QR이 성겨서 인식률도 올라갑니다.

### ⑨ 브라우저 호환 — **스파이크로 검증 완료 (2026-07-13)**

**결과: 🟢 통과.** iPhone (iOS 18.7 / Safari 26.5), **설치형 PWA(`standalone: true`)** 에서
카메라 기동(960×1280, 후면) + **QR·EAN-13 연속 디코딩 48건** 성공. **아키텍처 확정.**

- **iOS는 `BarcodeDetector` 미지원 → ZXing이 주력 디코더입니다.** 로컬 번들 필수 (CDN 금지 — 오프라인에서 죽음)
- Android·PC Chrome은 네이티브 `BarcodeDetector` 사용 → 훨씬 가벼움
- ⚠️ **ZXing 성능 제약 (iOS 실제 구현 시 필수):**
  1. **Web Worker로 디코딩 분리** — 메인 스레드 블로킹 방지
  2. **조준 프레임 안쪽만 크롭** 후 디코딩 — 픽셀 3~4배 절감
  3. **초당 ~10회로 스로틀** — `requestAnimationFrame`(60fps) 금지
  → 안 하면 **장서점검(수백 권 연속 스캔) 중 발열·배터리 소모가 심각**합니다
- ⚠️ **`toBlob('image/webp')`가 조용히 PNG로 폴백**함 → 반환된 `blob.type` 확인 후 JPEG 폴백

### ⑪ 사람이 쓰는 시트와 앱이 쓰는 시트를 분리하세요

`Books`·`Students`·`Loans`는 **앱 전용**입니다. 사서가 여기에 수기로 대출을 적는 건 **불가능합니다**
(필드 9개 · 시트 3개 · 날짜 계산 · 행 번호 암산). 그리고 앱이 죽으면 재동기화 버튼도 못 누릅니다.

→ **`수기입력` 시트 하나만 사서가 씁니다** (구분 · 도서번호 · 학생 · 처리자, 4개 필드).
→ `현황`·`찾기`는 수식 전용 읽기 시트. → 나머지는 **보호를 걸어 사서가 편집할 수 없게** 합니다.
→ 이렇게 하면 **②의 정렬 사고가 물리적으로 불가능해집니다.** 경고문보다 훨씬 강력합니다.
→ 앱 복귀 시 `admin.absorbManual()`이 흡수합니다. **상세: `docs/PATCH_SPEC.md` P3 (구 SCHEMA §2.14는 legacy/)**

### ⑩ GAS 6분 실행 제한

대량 작업(일괄 등록·진급·통계)은 **200건씩 배치 + `PropertiesService`에 커서 저장 + 클라이언트 재호출**.

## 3. 확정된 결정

### 아키텍처
```
GitHub Pages (프론트, PWA)  →  GAS 웹앱 (JSON API)  →  Google Sheets (DB)
                            ↘  Cloudflare Worker (CORS 프록시, 필요 시)
```
- GAS 코드도 **clasp로 git 관리** (GAS는 롤백이 사실상 없음)
- GitHub: 기존 계정 + **무료 Organization** 생성 (ToS: 1인 = 무료 계정 1개)
- 무료 Pages = 공개 레포. 비공개 원하면 Cloudflare Pages/Netlify

### 계정
- **인가 학교 아님** → 무료 Workspace 불가 → **개인 Gmail 티어** (일 90분)
- **문제없음.** 실사용 하루 20~30분. 사람 손이 서버보다 느림
- ⚠️ **프로젝트 전용 Gmail을 새로 만들고 기관이 소유.** 개인 계정 사용 금지 (인수인계·계정 잠금 리스크)
- ⚠️ 버그 폭주가 유일한 진짜 위험 → 실행 로그 + 재시도 상한

### 성능 3원칙 (5,000권 대응)
1. **도서ID = 행 번호** → O(1) 조회
2. **활성/이력 분리** — `Loans`(현재 대출중만, 수백 행) + `History_YYYY`(append-only)
3. **카탈로그 미러** — 전체 도서 JSON → Drive → 클라이언트 IndexedDB. **검색은 100% 클라이언트에서** (즉시·오프라인·할당량 0)
- **ISBN 조회는 브라우저에서** (GAS 아님) → 실행시간 1/5, UrlFetch 소모 0

### QR
- **책 QR = URL**: `https://lib.○○.kr/b/0005001` (도서ID 6자리 + Luhn 체크문자 1자리)
- **학생 QR = 불투명 ID만. URL 금지** (학생증이 공개 페이지로 연결되면 안 됨)
- QR 안에 책 정보 없음. **주소만.** 정보는 시트에
- 스캔 경로 2개: **앱 내 스캐너**(URL 벗겨서 ID만) + **폰 기본 카메라**(학생 셀프 대출)
- **같은 QR, 역할별 다른 응답** — 서버에서 분기. 비로그인은 "대출중(7/20 반납예정)"까지만, **대출자 이름 절대 노출 금지**
- ⚠️ 학생증 QR로 **로그인시키지 말 것** (QR은 신원 표시일 뿐 인증이 아님)

#### 🚫 기각된 대안: "ISBN 바코드를 관리 ID로 쓰기"

검토했고 **기각**했습니다. 다시 제안하지 마세요.

1. **ISBN은 "한 권"이 아니라 "책의 종류"입니다.** 같은 책 3권의 ISBN이 전부 같습니다 → 누가 어느 권을 빌렸는지, 어느 권을 폐기할지 특정 불가. **5,000권 규모에 복본은 반드시 존재합니다.** 세상의 모든 도서관이 ISBN이 인쇄돼 있는데도 자기 라벨을 붙이는 이유입니다.
2. **10~20%는 ISBN이 아예 없음** (기증본·구간·전집·원서) → 어차피 별도 번호 체계가 필요 → 시스템이 두 개가 됨
3. **1D 바코드(EAN-13)는 폰·웹캠으로 읽기 어렵습니다.** QR은 오류정정 30%, 각도 무관, 초점에 관대. **전용 스캐너가 없는 이 프로젝트에선 QR이 압도적으로 유리합니다**
4. 부가기호(ISBN 옆 5자리 바코드) 오독, 위치 제각각, 비닐·라벨에 가려짐
5. ISBN은 숫자일 뿐 → **URL 트릭 상실** (기본 카메라로 찍으면 책 소개가 열리는 설계가 무너짐)

**ISBN의 역할은 "등록 시 서지 자동입력" 한 번뿐입니다.** 그 이후 평생 우리 QR을 씁니다.
청구기호 라벨은 어차피 붙여야 하므로(서가 배열), **거기에 QR을 같이 인쇄하면 추가 비용은 0입니다.**

### 권한 4등급
| 등급 | 인증 | 핵심 |
|---|---|---|
| 관리자 | Google | 전부 + 설정·계정·일괄처리 |
| 사서·교사 | Google | 도서·학생 관리, 대출·반납, 통계 |
| **스테이션** | **기기 토큰** | 대출·반납·연장만. **학생 목록 검색 불가** |
| 학생 | 학번 + PIN | 셀프 대출, 본인 기록. **반납 불가** |

- **스테이션 = 기기 등록 방식.** 도우미 학생에게 계정 만들지 않음. 사서가 6자리 코드 발급 → 기기 토큰 저장 → 언제든 비활성화. 매년 도우미가 바뀌어도 기기만 등록/해제
- 스테이션은 **방금 스캔한 학생만 순간 표시.** 목록 검색 불가 (도우미가 친구 기록 뒤지는 것 방지)
- **다른 학생을 조회하는 API 자체를 만들지 말 것**
- **랭킹·한줄평은 닉네임으로** (실명 금지)
- PIN은 **해시 저장**. `임시PIN` 컬럼으로 사서 수기 발급 지원 (앱이 흡수 후 자동 삭제)

### UI/UX

**셸(Shell) / 뷰(View) 분리 — 이 프로젝트에서 가장 중요한 구조 규칙**

- **뷰** = 실제 기능 (대출 화면, 검색 화면, 도서등록 화면)
- **셸** = 뷰를 어떻게 배치·전환하는가 (데스크톱 = 창, 모바일 = 풀스크린 스택)
- 🔴 **뷰는 자기가 어디에 있는지 몰라야 합니다.** 타이틀바·닫기 버튼·리사이즈 핸들을 뷰 안에 넣지 마세요. 셸이 씌웁니다.
- 이 규칙 하나로 데스크톱↔모바일이 뷰 재작성 없이 됩니다. **v1의 탭 UI → v2의 윈도우 UI 전환도 뷰를 한 줄도 안 고치고 됩니다.**

**v1은 윈도우 셸을 만들지 않습니다.** 상주 스캐너 + 탭 + 내용 패널로 시작 (드래그·리사이즈·z-index 껍데기만 생략). **CameraService·scanBus·포커스 라우팅은 v1부터 그대로 만듭니다.**

**윈도우 컨트롤 (v2)**
- 최대화 · 닫기 · 최소화 · **새로고침** · **📌핀** 제공
- ⚠️ **"새로고침" = 데이터 재조회. 컴포넌트 재마운트가 아님** (폼 입력·스크롤 날아감)
- ⚠️ **📌핀은 최대 1개**, 걸려 있으면 화면 상단에 상시 배너 ("📋 장서점검이 스캔을 받는 중"). 없으면 대출하려던 스캔이 장서점검으로 들어갑니다
- 최소화된 창은 포커스가 없으므로 스캔을 받지 않음 (핀 제외)

**🔴 CameraService는 윈도우 시스템 바깥에 삽니다**
스캐너 창은 스트림의 **미리보기일 뿐**입니다. 창을 닫거나 최소화해도 카메라는 계속 돕니다. 이 분리로 "`<video>` 언마운트 → 스트림 재시작" 함정이 사라집니다.

**🔴 브라우저 F5로 오프라인 큐가 증발합니다**
큐를 메모리에 두지 마세요. **IndexedDB 필수.** 작성 중인 도서등록 폼도 자동 임시저장.

- **대출/반납은 하나의 뷰로 통합.** 책을 스캔하면 상태로 자동 판단 (`대출중`→반납, `보관중`→"누가 빌리나요?")
- **스캔 성공 시 소리 + 초록 플래시 필수** (화면 안 보고 작업함)
- **PWA** (Pages 호스팅이므로 가능). 오프라인 우선순위: 장서점검 > 대출/반납 큐 > 검색. 예약·설정·계정은 **온라인 전용**
- 라이브러리 (v2): **react-rnd** 또는 **WinBox.js**. 직접 만들지 말 것. `transform: translate3d` (top/left 금지), 단일 포털 루트

### 도서 등록 (0단계 = 프로젝트의 8할)
맨땅 5,000권 ≈ **85시간**. "다 정리하고 오픈"은 실패합니다.

> **전략: 점진적 등록.** 미등록 상태로 오픈하고, 학생이 등록 안 된 책을 가져오면 **그 자리에서 30초 등록 후 대출.** 인기 책부터 자연스럽게 소화됩니다.

**등록 흐름 (두 경로 모두 30초대):**
```
[ISBN 스캔]
  ├─ 성공(80~90%) → 서지 자동완성 + API 표지 → 사서 확인 → 저장
  └─ 없음(10~20%) → [표지 촬영] → AI로 서지 추출 → 사서 보정 → 저장
                      └─ [사진 없이 진행] → 수동 입력 + 촬영대기 큐
```
- ISBN 조회 순서: **국립중앙도서관**(광범위) → **알라딘**(표지 + **페이지수**) → 수동
- ISBN 없는 책: 기증본, 90년대 이전, 전집, 원서. **수동 입력 화면 필수** (없으면 프로젝트가 멈춤)
- **촬영을 강제하지 말 것.** `촬영대기` 큐로 추적. 등록이 멈추는 것보다 표지 몇 장 빈 게 낫습니다
- 사진 압축: **브라우저에서** 긴 변 800px + WebP 80% (3~5MB → 40~80KB)
- 표지 파일명 = **도서ID** (`000500.webp`). API 표지도 자체 저장 (외부 직링크 금지)
- 촬영 시 **학생 얼굴·이름표가 프레임에 들어갑니다** → 가이드 프레임 + 사서 승인 후 공개
- 💡 **서점에 납품 명세서(ISBN·서명·저자·페이지수 엑셀, 가능하면 MARC) 요청** → CSV 일괄 등록으로 0단계 절반 절약

### 폐기
- **행 삭제 절대 금지.** `상태 = 폐기` + `폐기일` + `폐기사유`
- **번호 재사용 금지.** `000500`은 영구히 그 책의 것
- `분실` → `폐기` 2단계 (장서점검에서 미검출 → 분실 후보)
- ⚠️ **복본 주의**: 같은 책 3권은 각각 다른 ID. 반드시 **QR 스캔으로 특정** (서명으로 찾으면 엉뚱한 권을 죽임)
- 폐기해도 과거 대출 이력이 살아있음 → "『○○○』 (폐기됨)"으로 정상 표시

### 난이도
- **컬럼**(`난이도`, `권장학년`) + **교체 가능한 컬러 스티커**
- 5,000번 판정은 불가능 → **자동 초안**(페이지수 + KDC + 알라딘 권장연령) → 사서 보정
- **반납 시 학생이 탭: 😊쉬움 / 😐적당 / 😰어려움** → `체감난이도` 누적. 사서 판단보다 정확하고, 포인트 트리거로도 활용

### 데이터 무결성
- 모든 쓰기에 **`LockService`** (ID 발번도 락 안에서)
- **`admin.resync()`** — `Loans` 기준으로 `Books.상태` + `Students.현재대출수` 재계산. 수기 편집 후 필수
