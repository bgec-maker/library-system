# 31 · derived-bench — 파생 뷰 O(n²) 감사·실측

## 왜
절대 규칙 8(파생 뷰 O(n²) 금지)을 지금까지 리뷰 눈으로만 지켰다. 장서 5,000권+ 전제이므로
파생 계산(viz·report·catalog 미러)을 실측으로 검증해 둔다.

## 무엇
- vizData·reportData·dashboardData·catalog·choseong의 파생 계산 정적 감사(중첩 순회 패턴)
- 의심 지점은 esbuild로 추출해 5,000권·2만 대출 더미로 실측(기준: 저사양 CPU 감안 100ms/계산)
- 걸리면 Map 인덱스/사전 집계로 수정, 안 걸리면 감사 결과를 이 파일에 기록(가짜 성공 금지 —
  근거 숫자 남김)
## 완료 조건: 실측 숫자 기록 + 수정분 검증 통과

## 감사·실측 결과 (2026-07-17 야간, 5,000행 더미 · Node/V8)
- **정적 감사**: 클라이언트 파생 계산 전수 스캔 — 행 규모(n=장서)에 걸리는 중첩 선형 탐색 없음.
  catalog 미러 병합=Map upsert O(n) ✓ · 검색 인덱스=행당 1회 구축 ✓ · viz/report는 서버 일배치
  집계 소비(VIZ 원칙 ①)라 클라이언트 n이 작음 ✓. 셸/레지스트리의 find/some은 n≤10 상수 ✓.
- **실측**:
  | 계산 | 실측 | 판정 |
  |---|---|---|
  | DataTable 정렬 — 현행 localeCompare(옵션) | **284.0ms** | ❌ 기준기(4GB) 환산 초 단위 |
  | DataTable 정렬 — Collator 재사용(수정안) | **13.2ms (×21.5)** | ✅ 적용 |
  | DataTable 텍스트 필터(6열) | 3.6ms | ✅ |
  | 검색 초성 인덱스 구축(행당 1회) | 14.4ms | ✅ |
  | 키 입력당 검색 매칭 | 2.4ms | ✅ (완료 조건 100ms 내) |
- **수정 1건**: DataTable 정렬 비교자의 호출당 Collator 생성 → 모듈 상수 재사용(의미 동일).
  촉발 조건: 5,000행 catalog에서 열 정렬 클릭 — e2e catalog 정렬 스텝이 동작 동일성을 검증.
