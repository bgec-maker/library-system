# 도서관 관리 MVP

Google Sheets를 운영 DB로 사용하면서, 이후 웹·앱과 실제 관계형 DB로 이전하기 쉽게 만든 도서관 관리 MVP입니다.

## 구성 파일

- `도서관_관리_MVP.xlsx`: Google Sheets로 가져올 워크북
- `Code.gs`: 메뉴, 데이터 검증, 검색, 등록, 대출, 반납·분실, 연장, 예약, 연체료, 감사 로그, 무결성 점검
- `Sidebar.html`: 직원용 통합 사이드바 화면
- `appsscript.json`: Apps Script V8 런타임과 최소 권한 설정 예시

## 설치

1. `도서관_관리_MVP.xlsx`를 Google Drive에 업로드해 연 뒤, **파일 → Google 스프레드시트로 저장**을 선택합니다. 새로 생성된 네이티브 Google 스프레드시트에서 다음 단계를 계속합니다. Office 편집 모드에서는 Apps Script 메뉴가 없을 수 있습니다.
2. 새 Google 스프레드시트에서 **확장 프로그램 → Apps Script**를 엽니다.
3. 기본 `Code.gs`의 내용을 이 폴더의 `Code.gs` 내용으로 교체합니다.
4. **+ → HTML**을 눌러 파일 이름을 `Sidebar`로 만들고 `Sidebar.html` 내용을 붙여넣습니다.
5. **프로젝트 설정 → 'appsscript.json 매니페스트 파일 표시'**를 켠 뒤 제공된 `appsscript.json` 내용으로 교체합니다. 프로젝트 시간대와 사용자 식별 권한을 맞추기 위한 필수 단계입니다.
6. 개인 소유 파일은 **반드시 파일 소유자 계정**으로 편집기에서 `setupLibraryMvp` 함수를 1회 실행하고 권한을 승인합니다. 최초 실행자를 소유자가 아닌 계정으로 시도하면 직원 행을 만들기 전에 중단합니다. 공유 드라이브처럼 개별 소유자가 없는 파일은 최초 운영할 `ADMIN`이 실행합니다. 이 과정은 DB 시트에 실제 편집 보호도 설정합니다.
7. 스프레드시트를 새로고침하고 **📚 도서관 관리 → 사이드바 열기**를 선택합니다.
8. 파일 소유자 또는 관리자 한 명만 **📚 도서관 관리 → 관리 → 일일 관리 트리거 설치**를 1회 실행합니다. 설치형 트리거는 설치한 계정 권한으로 동작합니다.

## 권한과 DB 시트 보호

- `03_TITLES` 이후 DB 시트는 경고만 표시하는 보호가 아니라 **활성 `ADMIN` 이메일만 편집자로 허용하는 실제 보호**를 사용합니다. 도메인 전체 편집 허용도 끕니다.
- Google Sheets 파일 소유자는 제품 특성상 보호 편집 권한에서 제거할 수 없습니다. 따라서 이 패키지는 파일 소유자를 항상 활성 `ADMIN`으로 유지하도록 강제하고 소유자 계정의 강등·비활성화를 차단합니다. 공유 드라이브처럼 개별 소유자가 없는 파일은 이 예외가 없습니다.
- 보호 설정 중 현재 계정 확인, 활성 관리자 확인, 보호 적용 중 하나라도 실패하면 작업을 오류로 중단합니다. 경고 전용 보호로 조용히 낮추지 않습니다.
- 사이드바에서 직원을 등록·변경하면 모든 DB 시트의 보호 편집자 목록도 현재 활성 `ADMIN` 목록으로 즉시 동기화됩니다. 중간 시트에서 보호 적용이 실패하면 이미 바뀐 보호와 직원 데이터 모두 변경 전 상태로 원복하고 작업을 실패로 기록합니다.
- `14_STAFF_USERS`를 직접 편집하면 이 동기화와 감사 로그를 우회합니다. 부득이하게 직접 정정했다면 활성 관리자가 **관리 → DB 시트 보호(ADMIN 전용)**을 즉시 다시 실행하고, 오류가 나면 직원 목록과 각 DB 보호 편집자를 함께 점검하세요.
- 바운드 사이드바의 서버 함수는 사이드바를 연 **현재 사용자 권한**으로 실행됩니다. 따라서 DB 보호를 켠 상태에서는 `STAFF` 시트에 `LIBRARIAN`으로 등록되어 있어도 해당 계정의 저장·대출·반납 같은 쓰기 작업이 Google Sheets 보호에 의해 차단됩니다. 앱 역할은 시트 보호를 우회하는 권한이 아닙니다.
- 이 MVP의 안전한 기본 운영 방식은 **파일 소유자 또는 활성 `ADMIN`만 스프레드시트 편집자로 두고, 그 계정 한 명이 바운드 사이드바를 사용하는 단일 운영자 방식**입니다.
- 여러 직원이 동시에 사용해야 한다면 DB 스프레드시트를 직원에게 편집자로 공유하지 말고, 별도 Web App/API를 **소유자 또는 전용 서비스 계정 권한으로 실행**하도록 배포하세요. 사용자 인증과 `STAFF` 역할 검증은 서버에서 수행하고, UI와 DB 파일 접근을 분리하는 구조를 권장합니다. 이 배포 구조는 현재 바운드 사이드바 패키지에 포함되어 있지 않습니다.
- 사이드바에서 메뉴를 숨기는 처리와 `STAFF` 역할만으로는 완전한 보안 경계가 되지 않습니다. 스프레드시트·바운드 스크립트 편집 권한 자체를 최소화하세요.

## 권장 최초 운영 순서

1. 회원 등록
2. 도서와 최초 소장본 등록
3. 대출 → 반납·분실 → 연장 → 예약 흐름 시험
4. **관리 → 무결성 점검** 실행
5. 실제 운영 전 샘플 데이터가 있다면 삭제하고, Drive 버전 기록 또는 별도 사본으로 백업

## 구현된 기능

- 운영 현황: 활성 도서, 대출 가능, 대출 중, 오늘 반납, 연체, 예약, 회원
- 통합 검색: 제목, 저자, ISBN, 바코드, 회원번호, 회원명, 대출 ID
- 회원 등록과 회원번호 자동 발급
- 서지·저자·분류 관계 등록과 소장본 등록
- 대출: 상태, 만료, 한도, 연체, 예약 우선권, 중복 대출 검증
- 반납: 연체 일수·연체료 생성, 다음 예약자에게 소장본 자동 배정
- 분실: 진행 중 대출과 소장본을 `LOST`로 변경하고, 명시한 대체 비용으로 `REPLACEMENT` 부과금 생성
- 연장: 정책상 횟수와 다음 예약자 검증
- 예약: 즉시 배정 또는 대기열 등록, 취소, 수령 기한 만료
- 소장본 상태 변경과 파생 상태 복구
- 연체료 부분/완납 기록
- 회원 연락처 중복 검증, 개인 대출 한도 검증, 거래가 남은 회원의 비활성·정지·탈퇴 전환 차단
- 마지막 활성 `ADMIN`과 파일 소유자 `ADMIN`의 강등·비활성화 차단
- 요청 ID 기반 중복 실행 방지, ScriptLock 기반 동시성 제어
- 감사 로그, 알림 큐, 무결성 점검, 일일 관리 트리거

## 분실 처리

1. 사이드바의 **반납·연장 → 대출 도서 분실 처리**를 엽니다.
2. 진행 중인 대출 ID 또는 소장본 바코드를 입력합니다.
3. 대체 비용을 0 이상의 정수로 반드시 입력하고 필요하면 분실 사유를 남깁니다.
4. 처리하면 대출은 `LOST`, 소장본은 `LOST`가 됩니다. 대체 비용이 1원 이상이면 `REPLACEMENT` 부과금이 생성되고, 0원이면 부과금 행을 만들지 않습니다. 작업 내용은 감사 로그에 기록됩니다.

진행 중 대출이 있는 소장본은 **관리 → 소장본 상태 수동 변경**으로 분실 처리할 수 없습니다. 반드시 이 전용 흐름을 사용하세요.

## 수기입력 (GAS 장애 시 비상 대출/반납)

Apps Script 자체가 응답하지 않을 때(배포 오류·네트워크 두절 등)도 대출·반납을 기록해 두었다가 GAS가 복구되면 한 번에 흡수하는 비상 경로입니다. `22_MANUAL_ENTRY` 시트는 이 패키지에서 **유일하게 ADMIN 전용 보호를 걸지 않는 쓰기 시트**입니다 — 사서가 GAS 없이 셀에 직접 타이핑할 수 있어야 이 경로의 존재 이유가 성립하기 때문입니다. 최초 실행(최초 설정 또는 수기입력 흡수) 시 자동으로 만들어집니다.

1. **GAS가 죽었을 때**: `22_MANUAL_ENTRY` 시트를 열고 빈 행에 다음 8열을 채웁니다.

   `일시 | 구분(대출/반납) | barcode | 학생(학번 또는 이름) | 처리자 | 메모 | 처리상태 | 처리결과`

   - **구분**: `대출` 또는 `반납` 중 정확히 하나만 적습니다.
   - **barcode**: 소장본 바코드입니다.
   - **학생(학번 또는 이름)**: 학번(`school_no`) 또는 도서관 회원번호(`member_no`)를 정확히 적거나, 이름을 적습니다. 동명이인이 있을 수 있는 이름 대신 학번/회원번호를 적는 편이 안전합니다.
   - **처리상태·처리결과**: 비워 둡니다 — 아래 흡수 실행이 채웁니다.

2. **GAS가 복구되면**: **📚 도서관 관리 → 관리 → 수기입력 흡수**를 실행합니다. 미처리 행(처리상태가 비어 있는 행)을 위에서 아래 순서로 재생해 실제 대출/반납 처리(`checkout_`/`return_`, 기존 로직 그대로)를 적용합니다.

3. **결과 확인**: 각 행의 처리상태가 `완료` 또는 `오류`로 채워집니다.
   - `오류` 행 중 **입력 오류**(학번/이름을 찾을 수 없음·동명이인·barcode 누락 등)는 값을 고치고 처리상태·처리결과 칸을 지우면 다음 실행에서 다시 시도됩니다.
   - `오류` 행 중 **처리 자체가 거부된 경우**(이미 대출 중인 소장본·정지된 회원 등, 실제 대출/반납 로직이 거부)는 같은 행을 고쳐도 재시도가 차단됩니다(요청 ID 기반 중복 실행 방지가 실패한 요청도 그대로 남기기 때문 — 위 "주의사항"의 원리와 같습니다). 이런 행은 고치지 말고 **새 행에 다시 입력**하세요.

4. 같은 흡수를 두 번 실행해도 이미 `완료`로 표시된 행은 건너뛰므로 중복 대출/반납이 생기지 않습니다.

5. 웹앱 데스크톱 대시보드 상단에 "수기입력 미처리 N건"이 표시됩니다. GAS가 복구된 뒤 이 값이 0이 될 때까지 흡수를 반복하세요. (운영센터 시트 안에서 수식으로 미처리 건수를 보여주는 것은 이후 패치 범위입니다.)

> 이 절차는 `02_사용법` 시트에도 같은 내용으로 옮겨 적어 두세요 — 이 저장소의 `도서관_관리_MVP.xlsx`는 이후 여러 패치(예: `20_VIZ_CACHE`·`21_BOOK_CACHE`)로 스키마가 이미 갈라진 초기 부트스트랩 템플릿이라 `02_사용법` 탭 내용까지 이 파일 안에서 함께 갱신하지 않습니다. 실제 운영 중인 스프레드시트의 `02_사용법` 탭 편집은 이 문서를 참고해 사서/관리자가 직접 반영하세요.

## 정책값 입력과 0의 의미

정책값은 모두 **0 이상의 정수**만 허용합니다. 관리 폼에서 빈칸은 기존 값을 유지하며, 숫자 `0`은 빈칸이나 기본값 대체가 아니라 다음과 같은 명시적 설정입니다.

| 정책값 | `0`의 의미 |
|---|---|
| 최대 대출 권수 | 개인 대출 한도 덮어쓰기가 없는 회원의 신규 대출을 허용하지 않음 |
| 최대 연장 횟수 | 연장을 허용하지 않음 |
| 최대 예약 건수 | 신규 예약을 허용하지 않음 |
| 대출 일수 | 날짜를 더하지 않고 대출 처리일의 종료 시각을 반납 기한으로 사용 |
| 연장 일수 | 날짜를 더하지 않고 기존 반납 예정일과 오늘 중 늦은 기준일의 종료 시각을 사용 |
| 예약 보관 일수 | 날짜를 더하지 않고 예약 배정일의 종료 시각을 수령 기한으로 사용 |
| 일 연체료 | 자동 연체 부과금을 생성하지 않음 |

회원의 개인 대출 한도는 비워 두면 정책값을 사용하고, 덮어쓸 때는 1 이상의 정수만 허용합니다.

## DB 구조와 데이터 검증

- 첫 번째 시트는 운영 화면, 두 번째 시트는 사용법이며, `03_TITLES` 이후는 향후 웹 프론트와 관계형 DB 이전을 고려한 ID·관계 중심 테이블입니다.
- 최초 설정/스키마 확인은 각 DB 시트의 현재 최대 행까지 상태 코드 데이터 검증을 적용합니다. 이후 스크립트가 새 행을 추가할 때도 해당 행에 같은 검증 규칙을 적용하므로, 초기 범위를 넘어 추가된 데이터가 검증 없이 남지 않습니다.
- 회원 연락처를 변경할 때는 자기 자신을 제외한 다른 회원과 전화번호·이메일이 중복되는지 확인합니다. 개인 대출 한도는 빈칸 또는 양의 정수만 허용합니다.
- 진행 중 대출, 수령 준비(`READY`) 예약, 미납·부분 납부 부과금이 남아 있는 회원은 비활성·정지·탈퇴 상태로 바꿀 수 없습니다. 먼저 해당 거래를 정리하세요.
- 마지막 활성 `ADMIN`은 강등하거나 비활성화할 수 없습니다. 다른 활성 관리자를 먼저 등록해야 합니다.
- 파일 소유자는 다른 관리자가 있더라도 보호 편집 권한에서 제거할 수 없으므로 활성 `ADMIN` 상태를 유지해야 합니다.

## 한/영 전환 (사이드바 언어 설정)

사이드바 헤더 오른쪽의 `KO`/`EN` 버튼으로 화면 언어를 바꿉니다. 이 설정은 **Google 계정별로 따로 저장**됩니다(Apps Script `PropertiesService.getUserProperties()` — 시트 전체가 공유하는 저장소가 아니라 로그인한 계정마다 격리된 저장소입니다). 그래서 같은 스프레드시트를 서로 다른 브라우저·계정으로 열어 둔 두 사람이 동시에 다른 언어를 볼 수 있습니다(한 사람은 한국어, 다른 사람은 영어). 새로고침하거나 나중에 다시 열어도 마지막으로 고른 언어가 그대로 유지됩니다.

- 폼 라벨·버튼·빈 화면 문구 등 사이드바 자체의 고정 문구는 즉시 번역됩니다.
- 회원 유형·상태, 소장본 상태·상태(컨디션), 분류 코드처럼 `16_CODEBOOK`/`06_CATEGORIES`에서 오는 값은 영어 모드에서 그 시트의 `label_en`/`name_en` 컬럼 값으로 바뀝니다(값이 비어 있으면 한국어 라벨로 자동 폴백 — 빈 칸이 보이는 일은 없습니다).
- 서버 오류 메시지는 사이드바 안에 내장된 코드→영어 매핑표로 번역됩니다. 매핑에 없는(드문) 오류 코드는 서버가 원래 보내는 한국어 메시지를 그대로 보여줍니다 — 빈 메시지가 뜨는 일은 없습니다.
- 검색 결과·무결성 점검 결과의 세부 문장(예: "소장 3권 / 대출가능 1권", "2학년 3반 12번" 같은 조합 문구)은 이번 라운드에서는 번역하지 않았습니다. 이 문장들은 서버 함수(`search_`, 무결성 점검)가 이미 완성된 한국어 문장으로 조합해 내려주므로, 사이드바에서 부분 문자열만 골라 번역하려면 그 서버 함수 자체를 고쳐야 하는데 이번 패치 범위 밖입니다(자세한 판단 근거는 `docs/ASSUMPTIONS.md`의 "todo/22" 절 참고).

### 운영센터 영어 탭(`01_Console_EN`) — 수기 생성 필요

`refreshDashboard_()`는 이제 `01_운영센터`(한국어)와 `01_Console_EN`(영어) **두 시트**에 같은 값을 씁니다 — 사용자별 토글이 아니라 "각자 자기 언어 탭을 연다" 방식입니다(사용법 시트와 마찬가지로 여러 사람이 동시에 보는 공유 화면이라 시트 자체는 언어별로 나눠야 합니다). 다만 이 코딩 환경은 `도서관_관리_MVP.xlsx`의 서식·라벨 셀을 새로 그릴 수 없으므로(`02_사용법` 탭과 같은 제약 — 아래 참고), **`01_Console_EN` 탭은 아직 실제 스프레드시트에 없습니다.** 그동안 `refreshDashboard_()`는 이 탭이 없으면 조용히 건너뛰므로 기존 사용에는 영향이 없습니다.

영어 콘솔을 실제로 쓰려면 사서/관리자가 직접:

1. `01_운영센터` 탭을 복제해 이름을 정확히 `01_Console_EN`으로 바꿉니다(따옴표 없이, 대소문자 그대로).
2. 라벨 텍스트(제목·통계 이름 등, 셀 값 자체)만 영어로 고쳐 씁니다 — `B3`·`E3`·`J3`·`A6`~`K6`·`A21:G27`·`I21:L27` 등 **숫자·값이 쓰이는 셀 좌표와 서식은 절대 옮기거나 바꾸지 마세요**(`refreshDashboard_`가 이 좌표에 그대로 씁니다).
3. 다음 「현황 새로고침」부터 자동으로 두 탭 모두 최신 값이 채워집니다 — 코드 변경은 필요 없습니다.

## 주의사항

- `03_TITLES` 이후 DB 시트의 헤더명은 변경하지 마세요.
- 최초 설정을 실행한 첫 계정이 `ADMIN`이 됩니다. 이후 사용자는 관리자가 사이드바의 직원 계정 관리에서 등록해야 하며, 미등록·비활성 계정은 접근이 거부됩니다.
- DB 시트를 직접 편집하면 Apps Script 잠금, 서버 검증, 감사 로그를 우회합니다. 활성 `ADMIN`도 정정은 가능한 한 사이드바에서 처리하세요.
- 주민등록번호, 비밀번호, 결제 카드, API 키는 시트에 저장하지 마세요.
- Google 계정 이메일을 확인할 수 없으면 접근을 거부합니다. `appsscript.json`의 `userinfo.email` 범위를 적용하고 권한을 다시 승인하세요.
- `executeWrite_`를 거치는 모든 쓰기 API와 관리 작업은 도메인 행, 감사 로그, 알림 큐 변경을 하나의 보상 작업으로 기록해 실패 시 역순 원복합니다. 작업의 `STARTED`/`FAILED` 행과 실패 감사 기록은 원복 밖에 유지해 장애 추적에 사용합니다. 다만 Google Sheets에는 네이티브 다중 시트 트랜잭션이 없으므로 원복 실패가 명시되면 즉시 무결성 점검과 백업 대조를 수행하세요.
- 동시 사용자, 지점, 권한, 데이터량이 크게 늘면 현재 ID와 관계 구조를 유지해 실제 DB로 이전하세요.
